# シミュレーション評価系 設計案(ツール7〜9)

**ステータス: 提案段階(2026-07-07)。ユーザー承認前。実装着手禁止。**
承認後は DESIGN.md の他ツールと同じ流儀(検証済み事実の引用・実装後の追記)で運用する。

## 目的

「ランダムな2体の対面 + 不確定な技・控えを無限試行してビッグデータ化する」構想の
実現可能な形。素朴な全空間ランダムサンプリングは (1)組合せ爆発で表が埋まらない、
(2)ランダム操作の勝率は実戦勝率と相関しない、(3)不完全情報の信念モデルが要る、
の3つの壁で機能しないため、次の2つに置き換える:

- **全ポケモン → メタセット条件付け**: 相手空間を `meta_sets.json`(上位構築の実在セット)に
  限定し、空間を 10^20 → 数十×数十 に落とす。
- **ランダム操作 → 校正済みポリシー**: 両者を「貪欲+ルールのポリシーAI」が操作し、
  そのポリシーを**ユーザー(対戦歴16年)との不一致クイズで校正**して実戦に近づける。

### 最終的にできること

1. 自軍セット vs メタ上位セットの **1v1勝率行列**(残HP分布つき)
2. 「この技/この調整に変えるとメタ加重勝率が±X%」という**構築最適化の定量比較**
3. 見せ合い6体からの**選出提案**(選出3体×先発の期待勝率行列とmaximin/混合戦略)
4. (副産物)ポリシーAIとの選択比較による対面判断の言語化・振り返り

### 非目標(やらないこと)

- 強化学習・ニューラルネット評価関数(研究グレードの工数になるため)
- 人間超えの対戦AI。ポリシーは「行列の数値を意味あるものにする」ための道具であり、
  対戦そのものの代行品ではない
- 型の隠匿・読み合いの価値の数値化(v0は完全情報仮定。限界の節を参照)
- テラスタル・Zワザ・ダイマックス等、現行Reg M-Bに存在しない機構の評価
  (ただし将来の追加に備えた拡張点として設計に織り込む — 「機構の扱いと拡張性」参照)

## 機構の扱いと拡張性

**決定事項(2026-07-07, ユーザー)**: Reg M-B シングルバトルのため、v0 で有効な特殊機構は
**メガシンカのみ**。テラスタルは考慮しない。ただしChampionsは運営型タイトルであり、
新ポケモン・新アイテム・Zワザ・ダイマックス・テラスタルが後から追加され得る前提で、
以下の拡張点を最初から設ける:

1. **mechanics設定の一元化**: `research/data/sim_config.json` に
   `{ "mega": true, "tera": false, "zmove": false, "dynamax": false }` を置き、
   全ツール(ポリシー・行列・選出)がここだけを参照する。機構の解禁は
   フラグON + ポリシーへの機構固有ルール追加(例: テラスタイミング判断)のみで済ませ、
   ツール本体の構造変更を要しない。
2. **選択肢列挙はrequest JSON駆動**: ポリシーの合法手は「simのrequestが出す選択肢を
   mechanics設定でフィルタ」して得る(技4つをハードコードで列挙しない)。
   simが新機構の選択肢を出すようになっても、フラグOFFなら自動的に落ち、
   ONにすれば列挙に自然に現れる。
3. **meta_sets v2 の予約フィールド**: セットに `teraType` / `zCrystal` / `gigantamax` 等の
   機構フィールドが将来増えてもスキーマ変更ではなく任意フィールド追加で済む構造にする。
   ツールは未知フィールドを黙って落とさず、レポートの前提欄に表示する。
4. **行列のキャッシュキー**: `matchup_matrix.json` の `_meta` に mechanics設定を記録し、
   設定が変わったら行列を再生成する(policyVersion・meta_setsハッシュと同格の扱い)。
5. **新ポケモン・新アイテムはデータ更新のみで追随**: ツール側に種・アイテム・技を
   ハードコードしない(既存ツール6と同じ原則)。新要素は usage CSV と meta_sets.json の
   更新だけで全ツールに反映される(sim側の実装は upstream pull + `node build` で取得)。

## meta_sets.json v2(スキーマ拡張 — 全ツールの土台)

現行(v1)は1種1セット・全てClaude推定。拡張点:

- **1種複数セット + 重み**: `"garchomp": { "sets": [ { "weight": 0.6, ...現行フィールド },
  { "weight": 0.4, ... } ] }`。重みは「その種を見たときこの型である確率」の推定。
  行列・選出計算は重み付き平均で不完全情報を近似する(信念モデルの最小実装)。
- **`source` の区分を厳格化**: `推定(Claude)` / `ユーザー監修(日付)` / `実測(対戦ログ)` の
  3区分。監修済みセットの割合をレポートに常掲する。
- 後方互換: v1形式(単一セット)は weight 1.0 の1セットとして読む。

## ツール7: 1v1マッチアップ行列(`research/tools/matchup_matrix.js`)

### 実装記録 (2026-07-08, フェーズ7b)

- `research/tools/matchup_matrix.js` および `research/data/sim_config.json` を新規作成。
- ポリシーv0: 貪欲+ルール(OHKO優先 / 積み安全判定 / 回復HP閾値 / まもる連打抑制)。ダメージ推定は `calcMatchup` のプログラムAPIを流用、ペアごとに事前計算(バトルループ外)してターンごとのオーバーヘッドを排除。
- 重み定数は全てコード冒頭に明記(W_RANK=0.15, W_STATUS_SLEEP=0.30等)。校正未実施のClaude初期値。
- CLIオプション: `--n`(既定500)、`--team`、`--only <speciesId>`、`--out`、`--report`、`--seed-base`。
- スモーク実行コマンド: `node research/tools/matchup_matrix.js --n 50 --only garchomp`
- 全ペア本計算コマンド: `node research/tools/matchup_matrix.js --n 500 --report`
- 全ペア推定所要時間: 276ペア×500試行≈13.8万バトル。7aベンチ(148試合/秒)から約15分(シングルスレッド)。

### ポリシーv0.1 校正第1号 (2026-07-08)

8-4サニティチェック起点の初号校正: アーマーガア vs HBガブリアス対面で、ユーザー感覚(ラス1同士なら必勝)と行列v0の出力(44%)が乖離。根本原因は、ボディプレス(鉄壁後の理論最大)や鉄壁そのものの価値を、現行ポリシーが評価できていなかったため。修正内容: (1)ボディプレス等「使用者の防御でダメージ計算する技」を overrideOffensiveStat で機械的に判定し、積み技評価時の攻撃力ゲイン計算に反映(526-527行)、(2)各ランク補正計算箇所でも bodypress 特例(408/436/465行)、(3)ポリシー重み W_SETUP_SAFE 0.25→0.35、W_RECOVERY 0.20→0.25。結果: 該当ペア勝率 44%/86% → 100%/100%(n=200確認)。policyVersion v0.1。

### ポリシーv0.2 校正第2号 (2026-07-09)

8-4校正第2号: アシレーヌ vs ミミッキュ_s0 の勝率 25〜28% をユーザー感覚(≈50%)と比較して調査。preComp実測値(sparklingaria max=94, moonblast max=100, ミミッキュHP=132)を確認した結果、技選択にコードバグは存在しなかった。タイプ相性はムーンフォース・うたかたのアリアとも等倍(Fairy vs Ghost/Fairy = 1×)であり、ムーンフォース(95BP)がうたかたのアリア(90BP)より正しく優先される。ユーザー予想の「ムーンフォース半減」という前提が誤りであり、25〜28%は貪欲ポリシー同士における力学的に正確な値(ミミッキュが素早さ148で先制、ディスガイズでT1攻撃を無効化、命の玉プレイラフが重い)。実際のコードバグとして `estimateOppMaxDamage` が「最大ダメージ」の関数名に反して `info.mid`(平均)を使用していた点を修正し `info.max` に変更(保守的な安全条件評価のため)。回帰ケース全通過確認(コルヴ vs ガブ 100%、ガブ vs カバ 0〜11%、ガブ vs キュウコン 0%、バシャ vs マホイップ 0%)。policyVersion v0.2。

### バトル駆動

- `gen9championscustomgame` + `battle.makeChoices()` 直接駆動(damage_calc.js の流儀)。
  BattleStream は使わない(同期・高速・デバッグ容易を優先)。
- 1ペアにつき seed を変えて N 回(既定は速度ベンチ後に決定、目安500)実行し、
  勝率と残HP分布を集計。乱数(16乱数・急所・追加効果・命中)はsimが正しく処理する
  ものをそのまま使う(自前実装しない — DESIGN.md共通ルール)。

### ポリシーv0(貪欲+ルール)

両プレイヤーとも同一ポリシー。完全情報(相手のセットを知っている)仮定。

1. 合法手を列挙(技4つ。1v1では交代なし)。
2. 各手について1ターン後の盤面価値を静的推定:
   `価値 = Δ相手HP% − Δ自分HP% + w_rank×ランク変化 + w_status×状態異常付与`
   (ダメージ推定は `calcMatchup` のプログラムAPIを流用。重み w_* は校正対象で、
   初期値はClaudeが仮置きし全てレポートに明記)
3. 相手の応手は「相手も同じ貪欲評価で最良手を選ぶ」と仮定して1手読み
   (同時手番の厳密解=混合戦略Nashはv0では扱わない。クイズ校正で乖離が
   大きければ v1 で fictitious play による1ターン混合戦略を検討)。
4. ルールによる上書き(校正で増減する想定の初期セット):
   - 相手を確定1発で倒せる技があれば最速でそれ(同点なら命中率優先)
   - 積み技: 「積んでも次ターン相手の最大打点で落ちない」かつ「積むと確定数が
     縮む」場合のみ選ぶ
   - 回復技: 残HPが相手の最大打点2発分を下回りそうなときに選ぶ
   - ねむり/まひはChampions固有の弱体化仕様(README既述: 眠り最大2ターン、
     まひ12.5%)を評価値に反映する

### 出力(`research/data/matchup_matrix.json` + mdレポート)

- ペアごと: `winA`(A側勝率)、`n`、`avgTurns`、勝者の残HP分布(10%刻みバケツ)、
  binomial 95%信頼区間。
- `_meta`: 生成日時、policyVersion、meta_sets のハッシュ、seed範囲。
  **meta_sets か policy が変わったら行列は再生成**(数時間のバッチ、ローカルCPUのみ)。
- レポートには「これは校正済みポリシー同士の勝率であり、上手い人間同士の勝率の
  近似にすぎない」を毎回明記。

## ツール8: ポリシー校正ループ(クイズ形式 — ユーザー経験の主注入点)

専用UIは作らない。**Claude Codeセッション内の対話**で行い、結果を
`research/data/calibration/*.jsonl` に記録する。3種のクイズフォーマット:

### 8-1. メタセット監修(最優先・実装不要・承認後すぐ可能)

- 形式: 「◯◯のメジャーな育成方針・型の分布は?」をClaudeが現行推定つきで提示し、
  ユーザーが訂正・重み付け。

  費用対効果が最も高い。既知の不確実ポイント(例: basculegion の物理/特殊混在の
  疑い — v1の source に「要ユーザー確認」と明記済み)から着手する。

### 8-2. 対面判断クイズ(ポリシー校正)

- 形式: 「この対面、両者の型は公開。盤面は◯◯(HP・ランク・状態・残りPP)。
  あなたの選択は?」— シミュレーション中の実局面からサンプリングして提示。
- **完全情報条件で行う**(両セット公開)。行列の前提と揃えるため。型読みの
  クイズは不完全情報版(将来)まで保留。
- 記録: 局面(再現可能なシリアライズ)、ユーザーの選択と理由、ポリシーの選択、
  一致/不一致。
- 不一致はポリシーのルール・重みの修正材料。**合意済みケースは回帰スイート化**し、
  ポリシー変更のたびに一致率が下がっていないか自動チェックする
  (ユーザーの判断が「テストケース」として資産になる)。

### 8-3. 選出クイズ(ツール9の校正)

- 形式: 「見せ合い。相手の6体はこれ(+上位構築CSV由来の実在の並び)。
  あなたの選出3体と先発は?」— モデルの選出提案と突き合わせる。

### 8-4. 出力サニティチェック(常設)

- 行列・レポートの数値で経験と食い違うものをユーザーが指摘
  (「この対面が7:3はおかしい」)。バグ・セット誤り・ポリシー欠陥の発見器として
  最も感度が高いチャネル。指摘は calibration ログに残し、原因調査の起点にする。

## ツール9: 選出提案・構築最適化(`research/tools/team_eval.js`)

### 9a. チェーンモデルによる選出行列

- 選出 = 3体選択×先発 = 6C3×3 = **60通り/側**。60×60 の期待勝率行列を作る。
- 各セルは1v1行列の**チェーン近似**で高速計算: 先発同士の1v1 → 勝者が残HPを
  引き継いで相手の次鋒と1v1 → …(残HP分布バケツを畳み込む)。
  交代・受け出しを無視する近似(限界の節参照)だが、60×60×相手セット重みを
  ミリ秒で回せるため、見せ合い画面を提示してその場で計算できる。
- 提案は maximin(相手の最善選出を仮定した保証勝率)を基本に、相手選出の
  経験的分布(8-3のクイズ・ユーザーの事前知識)があれば期待値版も併記。
- 相手6体は上位構築CSVの実在構築を使える(load_top_teams.js 再利用)。

### 9b. フル3v3シミュレーション + 構築最適化

- ポリシーに交代判断を追加(不利対面の静的判定 + 校正ルール)し、3v3を
  直接シミュレーション。チェーン近似の妥当性検証(両者の乖離を測る)を兼ねる。
- **構築最適化**: 自軍1体の技1枠/SP配分/持ち物を差し替えた変種セットを作り、
  対メタ行列の該当行だけ再計算(50ペア×N、数分)→ 採用率加重の総合勝率で
  ランキング。「この技を覚えさせると対メタ勝率+X%」が主出力。
  ツール6(move_value.js)の静的評価と相互検証できる(食い違いはどちらかの
  バグか、静的評価が拾えないシナジーの発見)。

## リスクと限界(レポートにも毎回明記する)

1. **ポリシー品質が全数値の上限**: 出てくる勝率は「校正済みgreedy同士」のもの。
   校正で人間の判断に近づけるが、深い読み・崩しのプランは表現できない。
2. **完全情報仮定(v0)**: 型を隠す価値・読み合いの価値はゼロと扱われる。
   複数セット+重みの平均化が最小限の緩和策。
3. **meta_sets の誤りは全数値に伝播**: だからこそ 8-1 の監修を最優先に置く。
4. **チェーン近似は受け回しを過小評価**: サイクル寄りの構築(自軍アーマーガア等)の
   評価が実際より辛く出る可能性。9bのフルシムとの乖離測定で定量化する。
5. 乱数由来の誤差: N=500 で勝率の95%CIは±4%程度。僅差の比較には N を増やす。

## 運用ルールとの整合

- データ源は自前シミュレーション+ユーザーの知識のみ。スクレイピングなし。
- meta_sets の更新は引き続き `source` 必須(監修/推定/実測の区別)。
- 行列・calibration ログは `research/data/` 配下。チーム変更に至った場合は
  従来どおり `teams/CHANGELOG.md` に記録。
- `node build` 後の dist/ を使う(既存ツールと同じ注意)。
