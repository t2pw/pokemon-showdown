# 分析ツール実装仕様書

このドキュメントは、パーティ相談の精度を上げるためのツール群の実装仕様。
Fable 5 が設計・技術検証を行い、実装は Opus / Sonnet セッションで行う想定。
**実装前にこのファイルと README.md を必ず読むこと。** 技術的事実はすべて
2026-07-06 時点のコードで検証済み(該当行を引用してある)。

## 前提: 検証済みの技術的事実

### フォーマット
- ID: `gen9championsbssregmb`
- 定義: `config/formats.ts` 「[Gen 9 Champions] BSS Reg M-B」 mod: `champions`,
  ruleset: `['Flat Rules', 'VGC Timer']`
- Flat Rules(`data/mods/champions/rulesets.ts` の `flatrules`)には `levelclausemod` が
  **含まれない** → 実数値はレベル非依存のChampions式で計算される。

### 実数値計算式(`data/mods/champions/scripts.ts` の `statModify`)
```
HP        = 種族値 + SP + 75
その他     = trunc(trunc((種族値 + SP + 20) × 110, 16bit) / 100)   # 性格↑
            (種族値 + SP + 20)                                     # 性格無補正
            trunc(trunc((種族値 + SP + 20) × 90, 16bit) / 100)     # 性格↓
```
- SP = Stat Points(Showdownチームデータでは `evs` フィールドに格納。1ステ最大32、合計66)
- 個体値は全31固定(validatorが強制)。この式では個体値・レベルは使われない。
- 例: ガブリアスS = (102+32+20)×1.1 = 169(ようき/SP32)

### その他の検証済み事実
- 全技PP最大20(`data/mods/champions/scripts.ts` init)
- **状態異常の仕様が本編と異なる**(`data/mods/champions/conditions.ts`、2026-07-06検証):
  - ねむり: 行動不能ターン数は `sample([2,3,3])-1` = **1ターン(確率1/3)か2ターン(確率2/3)、
    最大2ターン**。期待値 5/3 ≒ 1.67ターン(本編の1〜3ターン均等・期待値2より短い)
  - まひ: 行動不能は **1/8 (12.5%)**(本編25%の半分に弱体化)。素早さ半減は本編どおり
  - こおり: カウンタ3ターン+毎ターン1/4で解除(本編の「純粋に毎ターン20%解凍」と異なる)
- validator CLI: `node pokemon-showdown validate-team gen9championsbssregmb < team.txt`
  (stdin にエクスポート形式のチーム、exit 0 = 合法。要 `node build` 済みの dist/)
- ダメージ計算の本体: `sim/battle-actions.ts` の `getDamage(source, target, move)`
  (乱数ロールは `battle.actions.randomizer` 経由 → 差し替えれば16乱数を列挙できる)

## ツール1: 上位構築データローダー (`research/tools/load_top_teams.py` または `.ts`)

**目的**: `research/usage/s{N}_single_ranked_teams_YYYY-MM-DD.csv`(上位構築の生データ)の
最新ファイルを読み、集計統計を導出する共通部品。列構成の実物は README.md 参照
(2026-07-06のS2データ、222構築で確認済み)。

仕様:
1. `research/usage/` から最新日付のCSVを選ぶ(ファイル名の日付でソート)。
   BOM付きUTF-8のため `utf-8-sig` で読む。
2. 1行 = 1構築(順位・レート + 6スロット)。スロット列 `ポケモン_i` / `フォルム_i` /
   `持ち物_i` からポケモンごとのレコードに正規化する。フォルム(例: ウォッシュロトム、
   ヒスイのすがた)は種の同定に必須なので落とさない。
3. 日本語名→英名: 種は `jp_names_cache.json`、アイテムは
   `analysis_output/ja_item_map_raw.json` と `analysis_output/mega_stone_map.json`。
   未知の名前は警告を出して残す(黙って落とさない)。
4. 導出統計(関数として提供):
   - 採用率: 種ごとの登場構築数 / 総構築数(降順)
   - 持ち物分布: 種ごとのアイテム内訳(メガストーン率 = メガ運用率の推定になる)
   - 同時採用率: 種ペアの共起構築数(上位N種に限定して良い)
   - レート帯フィルタ: 例「レート2400以上のみ」で上記を再集計
5. CSVが1つもない場合は「usage/ にCSVを置いてください」と明示エラー。
6. 注意: これは「上位構築内の採用率」でありラダー全体の使用率ではない。
   出力・レポートには必ず「上位222構築中」のような母数を明記する。

## ツール2: ダメージ計算 (`research/tools/damage_calc.js`) — 実装済み(2026-07-06)

**目的**: Champions独自の実数値・アイテム・特性・場の状態を正確に反映した
ダメージ計算。一般のダメージ計算サイトはChampionsに対応していないため必須。

**方針**: 自前で式を再実装せず、simエンジンを直接使う(仕様変更に自動追従するため)。実装済み。

### 実装内容(仕様からの変更点含む)

- フォーマットは `gen9championsbssregmb` ではなく **`gen9championscustomgame`**
  (`[Gen 9 Champions] Custom Game`, `config/formats.ts` L318)を使用。
  理由: 同じ `champions` mod(実数値式・PP上限20は共通)でありながら、
  Reg M-B の Flat Rules が要求する Team Preview の Bring 6 / Min Team Size = 6 を
  回避でき、1体 vs 1体の純粋な計算に必要な最小構成(各1体のチーム)で足りる。
  Custom Game 側にも Team Preview 自体はあるため、`battle.makeChoices('team 1', 'team 1')`
  を1回呼んでから `battle.p1.active[0]` / `battle.p2.active[0]` を取得する。
- 16乱数は `battle.randomizer` を差し替えて実現(`baseDamage => trunc(trunc(baseDamage*i)/100)`,
  i=100..85)。急所判定の使い回しバグを避けるため、乱数ごとに
  `dex.getActiveMove(moveName)` で新しい ActiveMove を生成し `willCrit` を明示的に
  true/false 固定する(`Dex.getActiveMove` は文字列/非ActiveMoveを渡すたびdeep cloneする
  ため、`moveHitData` が乱数ごとにリセットされる — `sim/pokemon.ts` の `getMoveHitData`
  参照)。
- レベルは Champions の実数値計算には影響しない(SP式にレベル項がない)が、
  ダメージ計算式自体は本編通りレベル依存(`(2*level/5+2)` 項)。Flat Rules の
  Adjust Level = 50 に合わせ、デフォルト50固定(`--level` で変更可)。
- CLI引数はDESIGN.md原案の自由記述文字列(`--attacker-set "Adamant, SP: 32 HP..."`)
  ではなく、明示的なフラグ形式にした(パース精度と保守性を優先)。
  `--attacker-sp 32,0,2,0,32,0`(HP,Atk,Def,SpA,SpD,Speの順、カンマ区切り、
  Championsの1ステ最大32・合計66を検証)。
- 天候(`--weather sand|rain|sun|snow`)、フィールド(`--terrain misty|electric|grassy|psychic`)、
  壁(`--screens reflect,lightscreen,auroraveil`)、ランク補正
  (`--attacker-boosts atk:2,spe:1` 形式)、急所(`--crit`)、状態異常
  (`--attacker-status brn`)、テラスタル(`--attacker-tera Fire`)に対応。
- 出力: 16乱数のダメージ配列、最小〜最大とHP%、確定/乱数N発(`--defender-hp` で
  満タン以外のHPを指定可)。`--json` で機械可読出力。

### 検証済みテストケース(2026-07-06、3件とも手計算と一致)

1. **無補正のステータス計算 + 通常ダメージ**: Garchomp(ようき, SP A32/S32) の
   じしん → Hippowdon(わんぱく, SP H32/D32/D2)。実数値(Atk182/Def115/Spe169、
   Hippowdon Def187/HP215)を手計算と照合し完全一致。ダメージ55〜66
   (トゥルーバリューでSTAB込み計算と一致)。
2. **メガシンカ + アイテム + 天候**: Blaziken-Mega(いじっぱり, SP A32/D2/S32,
   いのちのたま) の フレアドライブ → Corviknight(わんぱく, たべのこし)、
   にほんばれ下。メガ後の実数値(Atk233等)を手計算と照合し一致。
   鋼複合への効果抜群+天候+いのちのたまが重なり確定1発を確認。
3. **ランク補正 + 急所 + 壁**: Scizor(いじっぱり, SP A32/S32, いのちのたま,
   こうげき+2) の アイアンヘッド級技(テストは弾丸パンチ) → Skarmory
   (わんぱく, ぼうぎょ-2, リフレクター展開下)、急所固定。
   実数値を手計算と照合(Scizor Atk200・Skarmory Def211・HP172)し一致。
   急所は「相手の防御上昇」のみ無視し「相手の自傷的な下降」は無視しない
   仕様(`sim/battle-actions.ts` の `ignorePositiveDefensive` 分岐)を確認。

### 既知の制限

- 複数回攻撃技・追加効果・命中率・連続ターン(退化技等)は非対応(単発ダメージのみ)。
- Terastal・状態異常・壁は明示フラグで手動指定(自動のターン進行は行わない)。

### 修正(2026-07-06): calcRolls の消費アイテム復元

半減の実(チョプル等)は sim の `getDamage` 内の ModifyDamage で `eatItem()` を呼び
バトル状態を書き換えるため、修正前は16乱数のうち**1乱数目だけ半減され、2乱数目以降は
実が消費済み扱い**になっていた(メガミミロップのとび膝→チョプルブリジュラスで
108〜212という不正なレンジを観測)。`calcRolls` の各乱数の後に攻守双方のアイテムを
`setItem()` で復元する修正を実施。

**検証**: 修正後の同計算は91〜108で、実なし(182〜216)のちょうど半分と一致。
非消費アイテムの回帰確認として検証済みテストケース1(ガブ地震→カバ 55〜66)が
不変であることを確認。

## ツール3: 素早さライン表 (`research/tools/speed_tiers.js` + 出力 `research/speed_tiers.md`) — 実装済み(2026-07-06)

**目的**: Champions式の素早さ実数値表。加速バシャーモの抜き調整や
スカーフ判断に使う。

### 実装内容(仕様からの変更点含む)

- 実数値の計算式は `sim/battle.ts` の `trunc`(`num >>> 0`、bits指定時は `% 2**bits`)と
  `data/mods/champions/scripts.ts` の `statModify` をそのまま移植した
  `calcStat(base, sp, mod)` で計算(mod: 1=性格↑, 0=無補正, -1=性格↓)。
  `overflowstatmod` はChampionsのルールセットに含まれないため実装していない
  (`data/mods/champions/rulesets.ts` に当該ルール名の記述なしを確認済み)。
- 対象種は「`isNonstandard` が null の全種族」ではなく、**ツール1の採用率上位30体**
  (`adoptionRate` の上位、`unresolved:` 接頭辞のものは除外)を実際に使用。
  M-B使用可否フィルタより「実戦で見る種か」を優先する方が相談用途に直結するため。
  各種についてメガシンカ後フォルムが存在すれば(`species.otherFormes` から
  `isMega` を検索)直後の行として必ず追加する。
  メガ後フォルムの採用率欄は**ベース種と同じ値**を表示する(CSVはメガストーン所持を
  「ベース種+アイテム」で表現しており、メガ限定の採用数は取得できないため。
  メガ運用率の推定には別途ツール1の `itemDistribution` を使うこと)。
- 自軍チームは `research/teams/` 内で最終更新日時が最も新しい `.txt` を自動選択し、
  Showdownエクスポート形式を簡易パース(種族名・アイテム・性格・EVs行のみ、
  技構成等は無視)。アイテムがメガストーンで対応表(`item.megaStone`)にヒットすれば
  メガ後の種族値で実速度を計算する。
- スカーフ(×1.5)・積み技+1(×1.5)・+2(×2)の参考値は `Math.floor(実速度 * 倍率)` の
  概算(chain modifyや16bit補正は考慮しない、相談用途には十分な精度と判断)。
- 出力は `research/speed_tiers.md` に上書き生成。採用率上位表(SP0/SP32 x 性格3パターン、
  SP32(↑)降順)と自軍チーム表の2セクション構成。

### 検証済みテストケース(2026-07-06)

- ガブリアス(ようき, SP32): 表内 `169` — ツール2の検証テスト(手計算169)と一致。
- メガゲンガー(おくびょう, SP32): `200` — 種族値130+32+20=182、性格補正×1.10を
  16bit補正込みで計算し200(182×110=20020, %65536=20020, /100→trunc→200)。
- メガバシャーモ(いじっぱり, SP0): `120` — 種族値100+0+20=120(いじっぱりはSに
  補正なしのため無補正の式そのまま)。

### 使い方

```
node research/tools/speed_tiers.js
```
`research/speed_tiers.md` に書き出す(引数なし、再実行のたびに上書き)。

## ツール5: 相談ワークフローのスキル化 (`.claude/skills/party-check/SKILL.md`) — 実装済み(2026-07-06)

**目的**: 毎回の相談で同じ準備を確実に行う定型手順。ツール1〜3が動いてから作る。

## ツール6: 技価値評価 (`research/tools/move_value.js`) — 6c実装済み・全段階完了(2026-07-06)

**目的**: 「技スロット1枠に、よく使われる変化技を入れるか、範囲を広げる攻撃技を
入れるか」という構築上の頻出判断を、上位構築メタに対する具体的な数値で支援する。
将来的には対戦の振り返り(盤面評価)の部品にもなる。一般公開はしない個人用。

**ユースケース例**: アーマーガアの4枠目に ボディプレス(攻撃技)/てっぺき(積み)/
ブレイブバード(範囲) のどれを入れるか、上位222構築への効果を比較する。

### 前提リファクタ

- `damage_calc.js` に プログラムAPI `calcMatchup(config)` を追加する
  (attacker/defenderのセットオブジェクト・技・場を受け取り、runと同じresultオブジェクトを
  返す。既存CLIの挙動は変えない。バトル構築部分を `run` から関数に切り出すだけ)。

### 新規データファイル: `research/data/meta_sets.json`

CSVには種族+持ち物しかなく**技構成・性格・SP配分は含まれない**ため、上位脅威の
「典型セット」を人手で管理するファイルが必要。構造:

```json
{
  "garchomp": {
    "nature": "Jolly", "sp": {"hp": 2, "atk": 32, "spe": 32},
    "ability": "Rough Skin", "item": "focussash",
    "moves": ["Earthquake", "Scale Shot", "Stealth Rock", "Rock Tomb"],
    "mainPhysical": "Earthquake", "mainSpecial": null,
    "source": "推定(2026-07-06, Claude): 標準物理配分+CSV最頻アイテム"
  }
}
```

- 初期データはClaudeがシード(性格・SPは標準配分32/32/2、持ち物はツール1の
  `itemDistribution` 最頻値、技はchampions modのlearnset+ゲーム知識から推定)し、
  **全エントリに `source` で推定であることを明記**する。
- ユーザーがポケモンバトルデータベースの個別ページを手動で見て転記・修正するのは
  リーガルかつ最も正確な更新手段(自動アクセスは引き続き禁止)。
- レポート出力には必ず「どのセットを仮定したか」を含める(仮定が違えば数値も違うため)。

### 評価モデル(技カテゴリ別)

共通: 対象は ツール1採用率上位N体(デフォルト15)。自軍セットは `teams/` 最新
ファイルから読む。命中率は期待値換算(例: おにび85% → 効果×0.85)をデフォルトとし
`--no-accuracy` で無効化可。

**攻撃技(範囲技候補)**:
- 候補技 vs 上位N体それぞれの16乱数・確定数を計算し、**既存技構成での最良確定数と
  比較**する。「この技の追加で確定数が改善する相手」のリストが主出力。
- 例出力: 「テラバースト(候補) はミミッキュに乱2→確2改善、他14体は既存技と同等以下」

**おにび**:
- 物理型の上位脅威それぞれについて: (a)相手の主要物理技→自軍各メンバーへの
  ダメージが火傷で半減する量/ターン、(b)最大HP1/16の定数ダメージ/ターン。
- 主指標: **損益分岐ターン数** = 「自分が攻撃する代わりにおにびを打った機会損失
  (自分の最良技1発分)を、(a)+(b)の累積が上回るまでのターン数」。
- ほのおタイプ・こんじょう・ミストフィールド等の無効対象は明示的に除外列挙。

**でんじは**:
- 素早さ半減(gen9仕様)により**自軍各メンバーとの素早さ関係が逆転する上位脅威の
  リスト**(speed_tiers.jsの `calcStat` を再利用)。
- 加えて行動不能 **1/8 (12.5%)/ターン**の期待値(**Champions固有の弱体化**、本編の25%と
  異なる — `data/mods/champions/conditions.ts` 検証済み。テンポ源としては本編の半分の
  価値しかなく、主価値は素早さ半減の方)。でんき・じめんタイプ無効を明示。

**ステルスロック**:
- CSVの**実際の222構築の6体構成**を使い、構築ごとに「メンバー平均の岩相性
  ダメージ(最大HPの3.125%〜50%)」を計算 → 全構築の分布として出す。
- 主指標: 「上位構築の平均で、交代1回あたり敵HP合計のX%」+ 期待交代回数
  (`--switches`、デフォルト2)を掛けた試算。SRが特に刺さる/刺さらない構築の例も列挙。

**どくびし**:
- 222構築それぞれについて: 接地かつ毒が入るメンバーの割合、および吸収役
  (接地どくタイプ)の有無。「どくびしが機能する構築の割合」が主指標。

**つるぎのまい(積み技全般)**:
- 自軍の攻撃技を+2(わるだくみなら特攻+2)で再計算し、**確定数が変わる上位脅威の
  リスト**(例: 確2→確1になる相手が15体中9体)。
- 補助指標: 積むターンに相手の主要技を1発受けられるか(耐久チェック)。

**ひかりのかべ/リフレクター**:
- 上位脅威の主要技→自軍各メンバーのダメージを壁あり/なしで計算し、
  **確定数が変わる(=行動回数が1回増える)組み合わせのリスト**。

**確定追加効果つき攻撃技(がんせきふうじ・バークアウト・ひけん・ちえなみ等)**:
- 「攻撃技モデル + 追加効果分の変化技モデル」の合算として扱う(ハイブリッド評価)。
- ランク変化系は dex の move.secondaries から **chance:100 + boosts** を自動検出して
  ディスパッチする(がんせきふうじ/こごえるかぜ/マッドショット→素早さ-1は
  でんじはモデルの倍率変更版(×2/3)、バークアウト→特攻-1はおにびモデルの変形)。
- 設置技ライダーは自動検出できないため小さな対応表をハードコード:
  `ceaselessedge→spikes`(接地相手に1/8)、`stoneaxe→stealthrock`。
  設置分の価値はステルスロックモデルを流用。

**行動制約系 — ネットテンポモデル + 部分指標**:

共通枠組み「**ネットテンポ**」= E[相手の行動不能ターン数] − 1(自分がその技に費やす
ターン)。命中率を掛けて期待値化する。眠りの行動不能ターン数はChampions固有仕様
(上記の 期待値5/3、最大2)を使う。

- **ねむり技(さいみんじゅつ・ねむりごな・キノコのほうし等)** — 完全に数値化可能:
  - ネットテンポ = 命中率 × 5/3 − 1
  - さいみんじゅつ(命中60): 0.6×5/3−1 = **±0.00**(期待値ちょうどゼロ。外し40%の
    分散が大きく、外した1ターンを相手の起点にされるリスクを考えると期待値以下の価値)
  - ねむりごな(命中75): **+0.25**(くさタイプ・ぼうじん系は無効 — 該当する上位脅威を列挙)
  - キノコのほうし(命中100): **+0.67**
  - レポートには「1ターンの価値は盤面依存(起点作り中の1ターンと中立盤面の1ターンは
    等価でない)」という前提を明記する。
- **あくび** — 分岐モデルで数値化可能(相手の選択で分岐するがどちらも計算できる):
  - 居座り: 相手は次ターン終了時に眠る → ネットテンポ = 5/3−1 = **+0.67**
  - 交代: 相手は交代にターンを使う → ネットテンポ **±0** + **設置技ダメージ**
    (ステルスロックモデルと連動 — SRがあれば交代強制1回=敵HP数%に換算)
  - どちらに転んでも損しない構造だが、交代側の分岐は「相手が有利対面を作る権利を得る」
    コストを含む(数値化しない、定性注記)。無効条件(ふみん・エレキ/ミスト
    フィールド等)該当の上位脅威を列挙。
- **ちょうはつ/アンコール** — 攻守両方向の部分指標:
  - 攻め(自分が採用): meta_sets.json の技構成から「技4つ中の変化技数」を集計し、
    「ちょうはつで技の半分以上が封じられる上位脅威はX体(内訳リスト)」。
    アンコールは自分が先手を取れることが前提なので速度関係リストも併記。
  - **守り(相手に撃たれる側)**: 覚えるポケモンが限られるため、(a) learnset から
    「ちょうはつ/アンコールを覚え得る上位脅威」を機械的に列挙(dexデータなので確実)、
    (b) 自軍各メンバーの変化技依存度(技4つ中の変化技数)と突き合わせて
    「撃たれるときついのは自軍の誰か」を出す。
- **みちづれ**: 1:1交換技でテンポモデルに乗らない。成立前提の速度関係リストのみ提示、
  価値本体は定性コメント。
- この系統(特にちょうはつ/アンコールの圧力価値)はフェーズB(シミュレーションA/B)で
  補強する価値が最も高い領域としてレポートにも明記する。

### 6b実装内容(2026-07-06)

`move_value.js` に変化技のモデルディスパッチを追加。`dex.moves.get(candidate)` の
`category`/`status`/`target`/`boosts`/`secondaries` から自動的にモデルを振り分ける
(CLIのインターフェースは6aから変更なし、候補技に変化技を渡せるようになっただけ)。

- **おにび(`modelBurn`)**: 損益分岐ターン数 = 機会損失(自分の既存攻撃技の最良1発の平均ダメージ) ÷
  ((a)自軍6体平均の被ダメ軽減 + (b)残飯毒ダメージ)。(a)は `calcMatchup` を
  `attacker.status: 'brn'` あり/なしの2回呼びで実際にsim側に半減させて差分を取る方式にした
  (火傷倍率を自前で再実装しない、DESIGN.md冒頭の禁止事項に準拠)。**検証**: Hippowdon→
  Blaziken-MegaのじしんをCLIで直接 `--attacker-status brn` あり/なしで実行し、
  180〜152(通常)→90〜76(火傷)とちょうど半分になることを確認(sim側の火傷処理が
  正しく効いていることの検証、モデル側の平均化ロジックはこの上に乗っているだけ)。
  ほのおタイプ/特性Water Bubble・Thermal Exchangeは無効として除外、
  特性Guts/Flare Boostは「逆効果」として警告表示。
- **でんじは(`modelSpeedControl`、`fullParalysis: true`)**: まひによる素早さ半減
  (×0.5)と、自軍6体との速度関係が逆転する相手を一覧化。1/8(12.5%)の行動不能率は
  README/DESIGN.md既出の定数として文中に明記(独立検証済みのため再計算はしない)。
  でんき/じめんタイプ・特性Limberを無効として除外。
  - **逸脱(スコープ判断)**: 素早さの比較はsimのバトルを介さず、`speed_tiers.js` の
    `calcStat`(検証済みのChampions実数値式)+ 標準のまひ50%/ランク段階倍率を直接計算した。
    まひによる速度半減・ランク段階倍率はChampions固有仕様ではなく本編共通仕様
    (README「素早さ半減は本編どおり」と明記済み)であり、比較専用の参考値算出のために
    毎回フルバトルを組むコストに見合わないと判断(`speed_tiers.js` のスカーフ参考値
    (`floor(実速度*1.5)`)と同じ位置づけ)。ダメージが絡む数値(確定数等)は
    引き続き全て `calcMatchup` 経由。
- **つるぎのまい(`modelSetup`)**: `move.target === 'self'` かつ正のboostsを持つ技
  (つるぎのまい/めいそう/りゅうのまい/からをやぶる/こうそくいどう系等、汎用的に自動判定)を
  検出し、`calcMatchup` の `attacker.boosts` に技のboostsをそのまま渡して確定数の変化を
  比較。積むターンの被弾チェック(相手の主力技のうちより痛い方1発を耐えられるか)も付記。
- **確定(chance:100)追加効果つき攻撃技**: `move.secondaries`(や単一の`move.secondary`)から
  `chance === 100` のエントリを自動検出し、`status`(brn/par)なら該当モデルを、
  `boosts`(spe/atk・spa/def・spd の低下)なら `describeOpponentStatDrop` 経由で
  でんじは変倍版/おにび変形/確定数改善チェックのいずれかをハイブリッド追記する
  (がんせきふうじ→でんじは変倍版、バークアウト/スナッチ相当→おにび変形、
  なやみのタネ相当→確定数改善チェック、で実際に動作確認済み)。
  自分がすでに撃っている攻撃技への「おまけ」効果という位置づけのため、
  おにび変形の機会損失は0として扱う(=常に「得」として出力される)。
  - **設置技ライダー(ひっさつしまい・ストーンアックス等)**: dexの`secondaries`からは
    設置効果を検出できない(`{}`の空オブジェクトが返るのみ)ため、DESIGN.md原案どおり
    `HAZARD_RIDER_MAP = {ceaselessedge: 'spikes', stoneaxe: 'stealthrock'}` を
    ハードコードした。ただし価値計算は6c(ステルスロックモデル)が前提のため、
    現時点では「6c未実装、価値は未計算」という注記のみを出す(6c完了時に接続する)。
  - **相手対象の純粋な変化技(スクリーチ・じゃれつく等)**: DESIGN.md原案は
    「攻撃技の確定追加効果」としてしか言及していなかったが、`move.target !== 'self'` かつ
    boostsが負の値を持つ純粋な変化技(スクリーチのぼうぎょ-2等)にも同じ
    `describeOpponentStatDrop` を直接適用できるようにディスパッチを拡張した
    (DESIGN.mdにない追加範囲。実装が自然に拡張できたため、レポートの有用性を優先した)。
- **行動制約系(ネットテンポモデル + 部分指標)**:
  - **眠り技(`modelSleep`)**: `ネットテンポ = 命中率 × 5/3 − 1` をそのまま実装。
    催眠術0.00・キノコのほうし+0.67等、DESIGN.md記載の数値と一致することを確認。
    粉技フラグ(`move.flags.powder`)を検出し、くさタイプ・特性ぼうじんを無効列挙。
  - **あくび(`modelYawn`)**: 居座り/交代の分岐値(+0.67 / ±0)を固定表示。
    特性(Insomnia/Vital Spirit/Sweet Veil/Comatose/Purifying Salt/Shields Down)による
    無効を列挙。場依存の無効条件(エレキ/ミストフィールド等)は定性注記に留めた
    (静的なmeta_sets.jsonデータからは判定できないため)。
  - **ちょうはつ(`modelTaunt`)**: 攻め側は上位脅威の変化技数(meta_sets.jsonの`moves`)、
    守り側は(a)`canLearnMove`(learnsetベース、baseSpeciesまで遡って判定)による
    習得可能種の列挙、(b)自軍6体の変化技依存度、の両方を実装。
  - **アンコール(`modelEncore`)**: 速度関係リスト(`effectiveSpeed`ヘルパー、
    でんじはと同じ参考値方式)+ 覚え得る上位脅威の列挙。
  - **みちづれ(`modelDestinyBond`)**: 速度関係リストのみ(定性コメント付き)。DESIGN.md
    どおり数値化はしていない。

**検証済みテストケース(2026-07-06、3件とも手計算/sim直接実行と一致)**:

1. **火傷の物理ダメージ半減(sim側の挙動そのもの)**: Hippowdon(わんぱく, SP H32/D32/D2)の
   じしん → Blaziken-Mega。通常16乱数152〜180、`--attacker-status brn` 指定で76〜90と
   ちょうど半分(端数の丸め込み以外は完全一致)。`modelBurn`の(a)算出はこの2回呼びの差分を
   使っているため、根拠となるsimの火傷処理自体が正しいことをこのテストで担保した。
2. **でんじは(まひ)の素早さ半減**: Archaludon(SP32(無)=105、`research/speed_tiers.md`の
   既存表と一致)に `modelSpeedControl(fullParalysis: true)` を適用すると
   `floor(105 * 0.5) = 52`。ツールの表示値と一致。
3. **がんせきふうじ相当(spe-1secondary)の素早さダウン**: Garchomp(ようき, SP32、
   素早さ169 — ツール2/3で既に検証済みの値)に `stage: -1`(×2/3)を適用すると
   `floor(169 * 2/3) = floor(112.67) = 112`。ツールの表示値と一致。

### 既知の制限(6b時点、6aの制限に追加)

- 素早さ比較系のモデル(でんじは/アンコール/みちづれ/攻撃技の確定spe-1効果)は前述のとおり
  simを介さない参考値計算(まひ50%・ランク段階倍率は標準仕様のため許容)。
- あくび/ちょうはつの「場依存の無効条件」(エレキ/ミストフィールド、ふみん等)は
  meta_sets.jsonが静的データのため機械的に判定できず、定性的な注意書きに留まる。
- 設置技ライダー(ひっさつしまい→まきびし、ストーンアックス→ステルスロック)は
  ハードコード対応表のみ実装済みで、価値計算は6c(ステルスロックモデル)待ち。
- 「相手対象の純粋な変化技」の一般ディスパッチは有用性のためDESIGN.md原案から拡張した
  箇所であり、想定外の変化技(命中率を下げる技等、boosts以外の効果)には対応していない
  (該当すれば「モデル未実装」として明示スキップされる)。

### 6c実装内容(2026-07-06)

DESIGN.md「ステルスロック」「どくびし」「ひかりのかべ/リフレクター」節と、設置技ライダーの
実配線を実装。ツール6(6a/6b/6c)は全段階完了。

- **ステルスロック(`modelStealthRock`)**: meta_setsではなく `research/usage/` の生CSV
  (222構築、`load_top_teams.loadTeams()`)を直接使う——DESIGN.md原案どおり
  「実際の6体構成」が必要なため(meta_setsは採用率上位15種のみで構築単位の情報を持たない)。
  被ダメ率は `0.125 * 2^(dex.getEffectiveness('Rock', types))` で算出
  (`dex.getEffectiveness` は複合タイプを配列で渡すと相性を自動合成してくれるため、
  タイプ相性表の自前実装なしで済んだ)。**検証**: Ferrothorn(くさ/はがね)が3.125%、
  Skarmory・Corviknight(はがね/ひこう)が12.5%、Charizard(ほのお/ひこう)が50%という
  「岩4倍弱点/等倍/4分の1」の周知の実例と一致することを確認(いずれも競技シーンで
  広く知られた値のため、独立した検証手段として採用)。
  出力は上位構築平均・期待交代回数を掛けた累計試算・特に刺さる/刺さらない構築の上位/下位3件。
- **どくびし(`modelToxicSpikes`)**: 同じくCSV生データを使用。接地判定は「ひこうタイプでない」、
  毒の通りやすさは「どく/はがねタイプでない」のみの簡易判定(ふゆう・ふうせん・めんえき等の
  個別要因は非対応、レポートに明記)。「機能する構築の割合」は「接地毒可メンバー1体以上 かつ
  吸収役(接地どくタイプ)なし」で定義(222構築中52.3%という結果)。
- **壁(`modelScreen`)**: meta_sets上位N体の`mainPhysical`/`mainSpecial`(リフレクター/
  ひかりのかべ/オーロラベールで対象を出し分け)→自軍6体それぞれへのダメージを
  `calcMatchup` の `field.screens` あり/なしで比較し、確定数が変わる組み合わせを列挙。
  オーロラベールは本編では霰/雪下限定の技だが、本ツールは「展開できた前提での価値」のみを
  試算する(注記あり)。
- **設置技ライダーの実配線**: 6bで用意した `HAZARD_RIDER_MAP`
  (`ceaselessedge→spikes`, `stoneaxe→stealthrock`)を、6cで実装した
  `modelStealthRock`/`modelSpikes`(まきびし1層・接地メンバーへ最大HP1/8の簡易モデル、
  タイプ相性は関与しない)に接続した。攻撃技本体の確定数比較の直後に、
  設置効果分の価値(上位構築平均)を追記する形。
- `--switches`(期待交代回数、デフォルト2)を実際にステルスロックモデルで使用するようになった
  (6bまでは予約フラグのみだった)。

**検証済みテストケース(2026-07-06、周知の実例3件と一致)**:

1. Ferrothorn(くさ/はがね複合、岩4分の1耐性): `stealthRockFraction` = 3.125%(1/32) —
   対戦者コミュニティで広く知られた「フェローチェはステロを最も気にしなくて良い代表例」の
   数値と一致。
2. Skarmory・Corviknight(はがね/ひこう複合、岩は2倍×0.5倍で等倍相殺): 12.5%(1/8) —
   「ひこう複合でもはがね持ちならステロは等倍で済む」という周知の事実と一致。
3. Charizard(ほのお/ひこう複合、岩4倍弱点): 50%(1/2) — 「バンギラスパーティの入れ替え性能の
   源泉」として広く知られる数値と一致。

### 既知の制限(6c時点)

- ステルスロック/どくびし/まきびしのモデルは `research/usage/` のCSVスナップショット
  (2026-07-06時点、222構築)に固定される。CSV更新時は再実行すれば自動で反映される。
- 接地判定・毒の通りやすさ判定はタイプのみの簡易判定(特性・アイテムによる個別例外は
  非対応、CSVにアイテム情報はあるが特性情報がないため機械的に判定できない)。
- 壁モデルはmeta_sets.jsonの`mainPhysical`/`mainSpecial`1本ずつのみ比較する
  (2つ目以降のコンボ技・複数回攻撃技は非対応、6a/6bと同じ制限を引き継ぐ)。
- まきびし(スパイク)は1層固定・タイプ相性なしの簡易モデル(2〜3層の重ね掛けや
  ねばりつくネット(スティッキーウェブ)は未実装)。

**やってはいけないこと**(全セッション共通):
- ダメージ式・実数値式の自前再実装(必ずsim経由 or 検証済みのcalcStatを使う)
- 個人サイトへの自動アクセス
- meta_sets.json の `source` なし追記(推定と実測の区別が消えるため)
